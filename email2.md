1. ON TOOLS
   Using familiar tools is fine — but the spirit of the sprint is to stretch. If you use Claude or ChatGPT every day, using it the same way you always do doesn't count as learning. Go deeper: try a feature you haven't touched, combine tools in a new way, or apply it to a type of problem you never have before. The bar is did you try something new — not did you use a brand new tool.

2. ON TEAM SETUP
   One shared idea. One prototype. One presentation. The point is to build something together — that's where the interesting conversations happen. You don't all need to be working in the same tool at the same time, but you should have one thing to show at the end — not three individual experiments loosely stitched together.

3. ON COLLABORATION
   Tools not supporting multi-user is fine. Divide the work — one person owns the build, another the testing, another the presentation slide. Use a shared doc or group chat to stay aligned. The collaboration is in the decisions, not the tool.

4. ON DEMO FORMAT
   Show something real. A live demo is ideal but not required — if your tool doesn't lend itself to that, screenshots or a short screen recording work fine. What's not acceptable is describing what you would have built. Show us the actual output, even if it's rough and unfinished.
